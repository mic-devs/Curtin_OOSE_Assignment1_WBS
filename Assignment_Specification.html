2 Problem  Description

Your  application will  be  a  tool   to   help  other  software  engineers perform  work  planning  and   effort   estimation.  It  will  both load  and  save  a   data   file  containing  a  hierarchical “work  breakdown  structure”  (WBS). ̢̡̡̨̡̡̦̤

For   our   purposes,  a  WBS  consists of  tasks,  some  of   which   are  broken  down  into other  tasks.  Every  task   has   an  ID  and  a short  description  (both  strings).   For   tasks  that  are  not broken  down  further,  they   also   have  an  optional  “effort estimate”  (a  positive  integer). ̢̡̡̨̡̡̦̤

2.1 WBS   Data   File

To  explain  this  further, consider  an  example  data   file: ̢̡̡̨̡̡̦̤

; A ; Market research ; 10
; B ; Prototyping ; 45
; C ; Game design
C ; D ; Identify core game concepts ; 15
C ; E ; Develop basic storyline ; 25
C ; F ; Develop game world and mechanics ; 55
C ; G ; Specify game characters and abilities ;
; H ; Game implementation
H ; I ; Artwork ; 90
H ; J ; Coding
J ; K ; Core game logic ;
J ; L ; Networking ; 35
J ; M ; Player interface ;
; N ; Testing
N ; O ; Functionality ;
N ; P ; Playability ;
N ; Q ; Rework ;
; R ; Release ;

The   letters  A-R  in  this example  are  chosen  arbitrarily—they   could   be  any  unique  strings. ̢̡̡̨̡̡̦̤

Each line  represents  a  task   in   the  WBS,  and  contains either  3  or  4   fields   separated  by  ;  (where whitespace  on  either  side   of   ;  is  optional  and should  be  discarded  by   your   application): ̢̡̡̨̡̡̦̤

  • The  1st  field,  which may  be  empty,  is   the   ID  of  the  parent task  (if  any). ̢̡̡̨̡̡̦̤
  • The  2nd   field   is  the  current  task’s ID. ̢̡̡̨̡̡̦̤
  • The  3rd  field  is   the   task’s  description,  which  may contain  any  characters  except   \n   and  ;. ̢̡̡̨̡̡̦̤
  • The  4th  field only  appears  on  tasks   that   are  not  broken  down. If  it  does  appear,   it   can  either  be  empty, or  a  positive  integer   representing   the  task’s  effort  estimate. ̢̡̡̨̡̡̦̤

Warning ̢̡̡̨̡̡̦̤

Your code  must  accept  any   file   in  this  format,  not just  the  specific  example   above!   It  is  advisable  to create  your  own  test   data,   to  ensure  it  can do  so. ̢̡̡̨̡̡̦̤

2.2 Loading  and  Saving

When   your   application  is  run,  it must  first  load  a   file   with  the  above  format. The  filename  is  to   be   provided  by  the  user as  a  command-line  argument,   and   your  program  must  accept any  filename.  (For  instance,   the   marker  may  supply  “thefile.txt” or  “./thefile.txt”  or  “/tmp/xyz”   or   “../../abc/def.ghi”,  and  your  program must  accept  any  of   them.) ̢̡̡̨̡̡̦̤

How to accept command-line filenames in Java

If   using  Gradle,  you  can run: ̢̡̡̨̡̡̦̤

$ ./gradlew run --args="thefile.txt"

The  string  "thefile.txt"  will   then   be  available  as  the first  element  of  the   args   array  parameter  in  your main()  method: ̢̡̡̨̡̡̦̤

public class MainClass
{
    public static void main(String[] args)
    {
        if(args.length != 1) { ... } // Error
        else
        {
            String filename = args[0];
            ...
        }
    }
    ...
}

How  to  accept   any   filename?  There’s  no  trick to  that,  except  to   not   manipulate  it.  For  instance, don’t  write  "dir/" + args[0]  (or   something   like  it).  Just  take args[0]  as-is. ̢̡̡̨̡̡̦̤

The  problem  of   supplying   the  correct  filename  is thus  the  user’s  problem.   If,   when  testing  your  program, you  get  a  "No such file or directory"   error,   then  the  problem  is probably  that,  as  a   user,   you  haven’t  supplied  the correct  filename,  or  the   correct   path  to  the  filename. ̢̡̡̨̡̡̦̤

When your  application  ends,  it   must   save  its  data  to the  same  file,  in   the   same  format,  overwriting  it. However,  your  application  may   (at   your  discretion): ̢̡̡̨̡̡̦̤

  • Skip  this  if the  data  hasn’t  actually   changed;   or ̢̡̡̨̡̡̦̤
  • Save  the  file  as soon  as  the  data   changes,   without  waiting  until  the end. ̢̡̡̨̡̡̦̤

Importantly,  your  application  must   also   (the  next  time  it is  run)  be  able   to   load  its  own  previously-saved data  file! ̢̡̡̨̡̡̦̤

2.3 Menu  Functionality

Your  application   must   show  a  console-based  menu to  the  user,  containing   the   following  options: ̢̡̡̨̡̡̦̤

  • Estimate  effort; ̢̡̡̨̡̡̦̤
  • Configure; ̢̡̡̨̡̡̦̤
  • Quit. ̢̡̡̨̡̡̦̤

The  “estimate effort”  option  is  central   to   the  application’s  purpose.  When selected,  it  should  do   the   following: ̢̡̡̨̡̡̦̤

  1. Ask  for  a  task ID  (which  may  be   the   ID  of  any  task); ̢̡̡̨̡̡̦̤

  2. If that  task  has  subtasks,   then   perform  the  following  steps for  each  subtask  instead   (recursing   further  into  any  subtask that  has  its  own   subtasks,   etc.): ̢̡̡̨̡̡̦̤

    ̢̡̡̨̡̡̦̤

  3. Ask  for  N  separate effort  estimates.  (N  should   be   3  by  default;  see the  “configure”  option  described   below.) ̢̡̡̨̡̡̦̤

    In   principle,  there  would  be N  separate  people  involved   in   the  estimation  process,  and they  would  each  enter   an   estimate,  one  by  one. This  is  like  a   modified   “planning  poker”  process.  However, for  simplicity,  we  won’t   restrict   the  allowable  estimates  in the  same  way  as   planning   poker. ̢̡̡̨̡̡̦̤

    Planning  poker  also  naturally relies  on  estimates  being   hidden   until  they  are  all simultaneously  revealed.  If  you   want,   you  can  use  the following  special  JDK  library   call   to  help  achieve  this: ̢̡̡̨̡̡̦̤

    String input = new String(System.console().readPassword());

    (Nothing will  appear  on-screen  as   the   user  types  their  input.) ̢̡̡̨̡̡̦̤

    ̢̡̡̨̡̡̦̤

  4. Show all  N  separate  effort   estimates   on-screen  (in  one  go). ̢̡̡̨̡̡̦̤

    ̢̡̡̨̡̡̦̤
  5. If any  two  (or  more)   effort   estimates  are  different,  apply the  selected  approach  for   reconciling   them  (see  the  “configure” option). ̢̡̡̨̡̡̦̤
  6. Record  the  new  effort   estimate   within  the  WBS. ̢̡̡̨̡̡̦̤

The  “configure” option  is  used  to   set   the  value  of  N (the  number  of  estimators),   as   well  as  the  approach for  reconciling  differing  estimates.   When   selected,  it  should  do the  following: ̢̡̡̨̡̡̦̤

  1. Ask  for  a   new   value  of  N,  which must  be  a  positive   integer.   (As  mentioned,  this  is 3  by  default.) ̢̡̡̨̡̡̦̤
  2. Ask  the   user   to  choose  a  new reconciliation  approach,  from  among   the   following:
    • Take  the  highest  estimate; ̢̡̡̨̡̡̦̤
    • Take the  median  estimate; ̢̡̡̨̡̡̦̤
    • Ask  the   estimators   to  discuss  among  themselves, and  request  a  single   revised   estimate.  (This  is  the default  choice,  before  the   “configure”   option  is  selected.) ̢̡̡̨̡̡̦̤
    ̢̡̡̨̡̡̦̤

2.4 Information  Display

Each time  the  menu  is   displayed,   your  application  must  first display  the  work  breakdown   structure   to  the  user.  When showing  the  WBS,  your   application   must  show  the  full tree  structure,  and  include   all   information  (except  parent  IDs). Probably  the  simplest  approach   is   to  indent  sub-tasks,  like this: ̢̡̡̨̡̡̦̤

A: Market research, effort = 10
B: Prototyping, effort = 45
C: Game design
  D: Identify core game concepts, effort = 15
  E: Develop basic storyline, effort = 25
  F: Develop game world and mechanics, effort = 55
  G: Specify game characters and abilities
H: Game implementation
  I: Artwork, effort = 90
  J: Coding
    K: Core game logic
    L: Networking, effort = 35
    M: Player interface
N: Testing
  O: Functionality
  P: Playability
  Q: Rework
R: Release

(You  need  not  adhere   to   this  exact  output  format, but  it  must  be   a   visual  tree  structure,  not just  a  dump  of   available   information.) ̢̡̡̨̡̡̦̤

Finally,  your  application  must also  output  an  effort   summary   that  includes: ̢̡̡̨̡̡̦̤

  • The  total  known effort,  being  the  sum   of   all  effort  estimates  across the  whole  WBS;  and ̢̡̡̨̡̡̦̤
  • The   number   of  unknown  tasks,  being the  count  of  all   tasks   that  don’t  have  subtasks, and  also  don’t  (yet)   have   effort  estimates. ̢̡̡̨̡̡̦̤

This  information  must appear  underneath  the  WBS   itself,   just  before  the  menu is  displayed.  For  instance: ̢̡̡̨̡̡̦̤

Total known effort = 275
Unknown tasks = 7

(In   our   running  example,  the  7 tasks  counted  as  “unknown   tasks”   are  G,  K,  M, O,  P,  Q  and   R.   Tasks  C,  H,  J and  N  are  not   counted,   because  they  are  broken down  into  subtasks.) ̢̡̡̨̡̡̦̤

3 Academic  Integrity

Please   see   the  Coding  and  Academic Integrity  Guidelines  on  Blackboard.   The   act  of  submitting  your work  to  Blackboard  will   be   taken  as  your  agreement with  these  principles. ̢̡̡̨̡̡̦̤

Curtin  also  provides general  advice  on  academic   integrity   at  academicintegrity.curtin.edu.au. ̢̡̡̨̡̡̦̤
